🏠 Executive Overview
🎯 What This Does (CTO Level)

This workflow automates the manual rebate processing pipeline for domain registrations and renewals. Previously a manual, error-prone process, it now runs on-demand via a Flock chat command or API call. Files uploaded to a Google Cloud Storage bucket are automatically ingested, validated against 13+ business rules, enriched with registry data from BigQuery, and a structured summary report is sent back to the team on Flock.

Business Impact Eliminates manual spreadsheet handling, enforces data quality before rebate disbursement, and provides instant visibility into processing status.

Node Details

✅ Validation Rules
How Validation Works All rows are loaded into a BigQuery temp table first. SQL UPDATE statements then flag invalid rows with an error code and status='ERROR'. Valid rows (validation_error IS NULL) proceed to domain enrichment. Multiple passes prevent false positives.
🟦 Phase 1 — JavaScript Pre-Processing (Before BigQuery Load)
#RuleWhat It DoesResult if Fails
P1 Domain Cleaning Strips protocol, path, port, trailing dots. Converts Unicode (IDN) to ASCII. Normalizes to lowercase. Row silently dropped (no domain = skip)
P2 Deduplication Within the same file, only the first occurrence of a domain is kept. Subsequent duplicates dropped before insert
P3 Date Parsing Handles Excel serial numbers, ISO format, DD/MM/YYYY, DD-MM-YYYY. Falls back to file's first row date, then today. Uses fallback date, never fails
P4 Domain Type Normalization Maps variants like "renewal" → "renew", "registration" → "new", "premium_new" → "premium new" Passes raw value if unrecognized (caught later)
🟧 Phase 2 — SQL Validations (In BigQuery Temp Table)
Note Rules are evaluated in order. Once a row has a validation_error set, subsequent rules skip it (AND validation_error IS NULL).
#Error CodeTrigger ConditionSeverity
S1 INCORRECT_DOMAIN_TYPE domain_type not in: new, renew, premium new, premium renew ERROR
S2 INVALID_DOMAIN_NAME NET.REG_DOMAIN(domain) returns NULL — not a valid registrable domain ERROR
S3 DUPLICATE_ENTRY Same domain appears more than once; all except the earliest created_at are flagged ERROR
S4 MISSING_DOMAIN domain column is NULL or empty string ERROR
S5 MISSING_DOMAIN_TYPE domain_type column is NULL or empty string ERROR
S6 MISSING_DATE date column is NULL ERROR
S7 MISSING_CLIENT client column is NULL or empty string ERROR
S8 MISSING_MBG_FOR_NEW_DOMAIN domain_type='new' and domain exists in newreg with mbg=NULL ERROR
S9 MISSING_MBG_FOR_RENEW_DOMAIN domain_type='renew' and domain exists in renews with mbg=NULL ERROR
S10 MISSING_MBG_FOR_PREMIUM_NEW_DOMAIN domain_type='premium new' and domain in premium_newreg with mbg=NULL ERROR
S11 MISSING_MBG_FOR_PREMIUM_RENEW_DOMAIN domain_type='premium renew' and domain in premium_renews with mbg=NULL ERROR
S12 MBG_DOMAIN domain mbg = 1 after enrichment (Money-Back Guarantee domain — not eligible for rebate) ERROR
🔧 Auto-Fix & Auto-Derive Rules (Applied After Validation)
TypeConditionAction
Auto-Upgrade domain_type='new' AND domain found in premium_newreg Sets domain_type = 'premium new'
Auto-Upgrade domain_type='renew' AND domain found in premium_renews Sets domain_type = 'premium renew'
Auto-Derive ANY non-empty promo_tag value (i.e. TRIM(COALESCE(promo_tag,'')) != '') Sets bundled = TRUE, else FALSE — used in ROI pricing join
Bundled Flag (Auto-Set) After validation, any row with a non-empty promo_tag value has bundled = TRUE set automatically (logic: IF(TRIM(COALESCE(promo_tag,'')) != '', TRUE, FALSE)). This flag is used in ROI pricing lookups to match bundle-specific pricing tiers.
🟥 Phase 3 — Post-Enrichment Validations
#Error CodeTrigger ConditionSeverity
E1 DATE_EXCEEDS_180_DAY_LIMIT createrenewdate is older than 180 days from today ERROR
E2 DOMAIN_MARKED_AS_DELETED domain_id is still NULL and domain found in cnic_dump.deleted ERROR
E3 DOMAIN_FOUND_IN_RESTORE domain_id is still NULL and domain found in cnic_dump.restore ERROR
E4 DOMAIN_NOT_FOUND_FOR_DOMAIN_TYPE domain_id is still NULL after all lookup steps ERROR
E5 MBG_DOMAIN COALESCE(mbg, 0) = 1 — domain is under Money-Back Guarantee, not eligible for rebate ERROR
E6 UNREALIZED_DOMAIN Domain has no effective_first price and rebate could not be calculated ERROR

📘 KT / Knowledge Transfer Runbook
Audience This section is written for new team members or engineers taking over maintenance. It covers how to trigger the workflow, how to read results, and what to do when things break.
1

How to Trigger the Workflow

Two ways to start rebate processing

Option A — Flock Chat Command

Send this exact message to the Flock channel connected to the webhook:

process rebates

The webhook also validates a secret token in the query parameter. This is configured in the Switch node and should match the token in your Flock webhook URL.

Option B — Manual Execute in n8n

Open the workflow in n8n and click the Execute Workflow button. This bypasses the Switch/token validation and goes straight to file processing.

Note Manual execution skips the Webhook → Switch path. Both paths merge at the HTTP Request (Flock start notification) node.
2

How to Upload Input Files

File format requirements and GCS upload

  1. Prepare your file as .xlsx or .csv
  2. Required columns: domain, domain_type, client
  3. Optional columns: date, promo_tag
  4. Upload the file to the GCS bucket: manual_rebate_process
  5. Trigger the workflow (see above)
Important The workflow processes ALL .xlsx/.csv files in the bucket at the time of execution. Remove old files from the bucket before running to avoid reprocessing stale data.
3

Reading the Flock Report

Understanding the output notification

After processing, a Flock message is sent like this:

✅ Rebate processing completed successfully Total rows: 120 Successful rows: 115 Failed rows: 5 Breakup: • SUCCESS: 115 • MISSING_MBG_FOR_NEW_DOMAIN: 3 • DOMAIN_NOT_FOUND_FOR_DOMAIN_TYPE: 2

Each error code tells you exactly what went wrong. See the Error Codes tab for remediation steps.

4

Querying Results in BigQuery

How to inspect processed data

-- See all successful rows SELECT * FROM `rebate_process_temp.rebate_cleaned_data` WHERE validation_error IS NULL; -- See all error rows with breakdown SELECT validation_error, COUNT(*) AS cnt FROM `rebate_process_temp.rebate_cleaned_data` WHERE validation_error IS NOT NULL GROUP BY validation_error ORDER BY cnt DESC; -- Inspect specific error SELECT domain, domain_type, client, date FROM `rebate_process_temp.rebate_cleaned_data` WHERE validation_error = 'DOMAIN_NOT_FOUND_FOR_DOMAIN_TYPE';
Caution The temp table is TRUNCATED and reloaded every time the workflow runs. If you need to preserve data, export it before re-triggering.

6

Common Maintenance Tasks

Day-to-day operational knowledge

Add a new validation rule ▼

Add a new UPDATE statement to the Validations Queries BigQuery node. Follow the pattern:

UPDATE `rebate_process_temp.rebate_cleaned_data` SET validation_error = 'YOUR_ERROR_CODE', status = 'ERROR', updated_at = CURRENT_TIMESTAMP() WHERE [your condition] AND validation_error IS NULL; -- always add this guard
Change the Flock notification channel ▼

Update the URL in both HTTP Request nodes (the start notification and the final report). The Flock webhook URL is the only thing that needs changing.

Add a new domain type ▼

Changes needed in 3 places:

  1. clean data JS node — add to cleanDomainType() function
  2. Validations Queries BQ node — add to the INCORRECT_DOMAIN_TYPE allowlist
  3. find domain in system BQ node — add an UPDATE block for the new type's source table
The workflow processed old files again ▼

This happens because the GCS bucket is not cleared between runs. The workflow picks up ALL .xlsx/.csv files currently in the manual_rebate_process bucket. Solution: delete processed files from the bucket, or add a post-processing step to move files to an archive folder.

⚠️ Error Code Reference
How to Use This Reference When you receive a Flock report with error codes, look up each code here to understand the root cause and how to fix the input data.
INCORRECT_DOMAIN_TYPE Invalid domain type value ▼
Phase
SQL Validation #1
Root Cause
The domain_type column has a value that isn't one of the 4 supported types after normalization
Fix
Correct the domain_type to: new, renew, premium new, or premium renew
Also Accepts
renewal, registration, create, premium_new, premium_renew
INVALID_DOMAIN_NAME Not a valid registrable domain ▼
Phase
SQL Validation #2
Root Cause
BigQuery's NET.REG_DOMAIN() function returned NULL — the domain is malformed or not a valid ICANN registrable domain
Fix
Check for typos, extra spaces, or junk characters. The domain must have a valid TLD (e.g. example.com, not "example")
DUPLICATE_ENTRY Domain appears more than once ▼
Phase
SQL Validation #3
Root Cause
The same domain name was submitted more than once across all files in the bucket. Only the first entry (by created_at) is kept.
Fix
Remove duplicate rows from your input file before uploading
MISSING_DOMAIN / MISSING_DOMAIN_TYPE / MISSING_DATE / MISSING_CLIENT Required field empty ▼
Phase
SQL Validations #4–7
Root Cause
A mandatory column is NULL or an empty string in the input row
Fix
Ensure all four required columns are filled in: domain, domain_type, client, date
MISSING_MBG_FOR_*_DOMAIN MBG field is NULL in registry ▼
Phase
SQL Validations #8–11
Root Cause
The domain was found in the registry table but the mbg (Money-Back Guarantee) field is NULL — rebate cannot be calculated without it
Fix
Update the MBG field in the relevant registry table (radix_data.newreg, renews, etc.) for this domain before re-running
Variants
MISSING_MBG_FOR_NEW_DOMAIN · MISSING_MBG_FOR_RENEW_DOMAIN · MISSING_MBG_FOR_PREMIUM_NEW_DOMAIN · MISSING_MBG_FOR_PREMIUM_RENEW_DOMAIN
DATE_EXCEEDS_180_DAY_LIMIT Registration/renewal too old ▼
Phase
Post-Enrichment Validation #1
Root Cause
The createrenewdate (from registry) is older than 180 days from today. Rebate window has expired.
Fix
This domain is no longer eligible for a rebate. Remove it from the claim file. If you believe this is an error, verify the actual registration/renewal date in BigQuery.
DOMAIN_NOT_FOUND_FOR_DOMAIN_TYPE Domain not in registry ▼
Phase
Post-Enrichment Validation #4
Root Cause
After checking all 4 registry tables (newreg, renews, premium_newreg, premium_renews), no matching record was found. The domain either doesn't exist or was submitted with the wrong domain_type.
Fix
Verify: (1) Is the domain registered in our system? (2) Is the domain_type correct? (3) Could it be in a different registry table?
DOMAIN_MARKED_AS_DELETED Domain has been deleted ▼
Phase
Post-Enrichment Validation #2
Root Cause
The domain was not found in active registry tables, but exists in cnic_dump.deleted — the domain was deleted/expired
Fix
Deleted domains are not eligible for rebates. Remove from claim file.
DOMAIN_FOUND_IN_RESTORE Domain is in restore/redemption state ▼
Phase
Post-Enrichment Validation #3
Root Cause
Domain was not found in active tables but exists in cnic_dump.restore — domain is in redemption/restore grace period
Fix
Domain is in a special state. Wait for restoration to complete or verify eligibility with the registry team.
MBG_DOMAIN Domain is under Money-Back Guarantee ▼
Phase
Post-Enrichment Validation #5
Root Cause
The mbg field returned from the registry table equals 1 — the domain is still within its Money-Back Guarantee window and is therefore not eligible for a rebate
Fix
Wait until the MBG period has expired before claiming the rebate for this domain, or verify with the registry team whether an exception applies
SQL Check
COALESCE(mbg, 0) = 1 AND validation_error IS NULL
UNREALIZED_DOMAIN No effective_first price — rebate cannot be calculated ▼
Phase
Post-Enrichment / Rebate Calculation
Root Cause
Domain matched in registry but no effective first-year pricing found in falcon.roi_records (and no CNIC or std-price fallback resolved it), so effective_first is NULL and rebate cannot be derived.
Fix
Check falcon.roi_records for the domain's TLD/registrar/period combination. Ensure pricing records exist. Re-run after adding pricing data.
